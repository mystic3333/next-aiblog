大多数用户是在两周后放弃图谱的。第一周觉得好酷，第二周发现图上那个毛线球什么都看不出来，然后就再也没打开过。

这篇文章不教你怎么建链接，假设你已经在用了。我们聊的是：让链接体系在三个月后仍然有用，而不是变成垃圾场。

## 结构笔记：给你的图谱装上骨架

大多数人犯的第一个错误：把所有笔记直接扔进同一个池子，然后期待图谱自动变有用。不会的。你需要结构笔记。

结构笔记不记录「内容」，它只记录「连接」。它是一张地图，而不是地图上的一个景点。

假设你在学 React，你可能有这些笔记：

- React useState 详解
- useEffect 生命周期
- React 自定义 Hook 模式
- React Server Component

如果全部打上 `#React` 标签，图谱上它们散落各处。如果全部双向链接到彼此，你会得到一个 20 个节点的毛线球。

正确做法：建一篇 `React 知识地图`（MOC, Map of Content），里面只有列表和链接：

```
# React 知识地图

## 核心概念
- [[React 核心渲染机制]]
- [[JSX 编译原理]]
- [[Virtual DOM 与 Reconciliation]]

## Hooks
- [[useState 详解]]
- [[useEffect 生命周期]]
- [[自定义 Hook 模式]]
```

然后，每篇子笔记只链接回这篇结构笔记，以及直接相关的兄弟笔记。你的图谱会出现一个清晰的星形结构——中心是 React 知识地图，外围是功能模块。

实操规则：

- 一个主题超过 5 篇子笔记，就该建一篇结构笔记
- 命名统一用 `XXX 知识地图` 或 `XXX MOC`，方便搜索
- 每周维护一次，5 分钟

## 链接 vs 提及：区分关系强度

Obsidian 在右侧面板里把链接和未链接的提及分开展示，但大多数人没利用好这个区分。

规则很简单：

- **双向链接** = 强关系。这篇笔记直接依赖那篇笔记的内容，或者两者共同支撑同一个论点。
- **纯文本提及** = 弱关系。只是顺带提到，不是核心依赖。

效果：三个月后你搜一篇旧笔记，强链接返回的是「真正相关」的东西，而不是所有出现过那个词的文件。

落地方式：在 Templater 模板里加一个 frontmatter 字段 `related`，每次建笔记时强制指定一篇核心关联笔记。其余的全部用纯文本引用。

## 局部图谱是你的日常工具

全局图谱是装饰品。局部图谱才是生产力。

进入一篇笔记后，`Cmd / Ctrl + Click` 图谱图标打开局部图谱。你每天应该用的就是这个。

三个高效用法：

**1. 追踪未链接的提及**

局部图谱里的灰色虚线节点，是你提到过但没建链接的地方。值得去看一眼——如果确实相关就建个双向链接。这是知识发现的核心机制。

**2. 路径查找**

在某个主题上卡住时，从当前节点的局部图谱出发，看哪些节点连接了它但不直接相关。跨领域连接往往是灵感的来源。

**3. 深度过滤**

在 filter 面板里设置仅显示路径深度 1-2 的节点。聚焦在当前笔记的直接相关材料上，不被全局图谱淹没。

## 什么时候该建链接

很多人的困惑：「什么程度才值得建一个链接？」

我的原则：

- **建新笔记时**：如果这篇笔记指向一个你未来会扩展的概念，建链接。如果只是提一下，不建。
- **阅读时**：发现一个概念可以支撑另一篇已有的笔记，回到那篇笔记建反向链接。
- **整理时**：每周一次，打开「未链接的提及」面板，批量处理。3 秒内判断不出是否该链接的，不建。

链接不是分类。分类用标签。链接是探索路径。

## 核心就一句话

双向链接最反直觉的地方：质量比数量重要一百倍。一张 30 个节点但结构清晰的图谱，比 3000 个节点互相乱连的图谱有用得多。

每篇新笔记建好后，花 10 秒问自己一句：「一年后来看这篇笔记，我最希望从哪个入口找到它？」然后把这个入口作为你的核心链接。
